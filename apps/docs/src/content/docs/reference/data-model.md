## The `employee` table

```ts
// packages/db/src/schema/employee.ts
import {
  pgTable, uuid, varchar, integer, date, timestamp, index, pgEnum,
} from "drizzle-orm/pg-core";

export const employmentType = pgEnum("employment_type", [
  "full_time", "part_time", "contract", "intern",
]);

export const employee = pgTable(
  "employee",
  {
    id: uuid("id").primaryKey().defaultRandom(),

    // Required by the assignment
    firstName: varchar("first_name", { length: 100 }).notNull(),
    lastName: varchar("last_name", { length: 100 }).notNull(),
    jobTitle: varchar("job_title", { length: 120 }).notNull(),
    countryCode: varchar("country_code", { length: 2 }).notNull(), // ISO 3166-1 alpha-2
    salary: integer("salary").notNull(), // minor units (cents), see "Storing money"

    // Meaningful additions for an HR manager
    currency: varchar("currency", { length: 3 }).notNull().default("USD"), // ISO 4217
    department: varchar("department", { length: 120 }),
    email: varchar("email", { length: 255 }).unique(),
    employmentType: employmentType("employment_type").notNull().default("full_time"),
    hireDate: date("hire_date"),

    createdAt: timestamp("created_at").defaultNow().notNull(),
    updatedAt: timestamp("updated_at").defaultNow().$onUpdate(() => new Date()).notNull(),
  },
  (t) => [
    index("employee_country_idx").on(t.countryCode),
    index("employee_country_job_idx").on(t.countryCode, t.jobTitle),
    index("employee_job_idx").on(t.jobTitle),
  ]
);
```

## Field-by-field rationale

| Field | Type | Why it exists |
| --- | --- | --- |
| `id` | `uuid` PK | Stable, non-guessable identifier; safe in URLs; no sequence contention during bulk seed. |
| `firstName` / `lastName` | `varchar` | Stored **separately**, not as one "full name" string — see below. |
| `jobTitle` | `varchar` | Required; the grouping dimension for the "avg salary for a job title in a country" insight. |
| `countryCode` | `char(2)` ISO | Required; normalized 2-letter code so grouping is exact (`"DE"`, never `"Germany"` vs `"germany"`). |
| `salary` | `integer` | Required; stored in **minor units**. See "Storing money". |
| `currency` | `char(3)` ISO 4217 | A salary number is meaningless without it; lets us label and (later) normalize. |
| `department` | `varchar?` | Enables a useful secondary breakdown for HR; nullable. |
| `email` | `varchar? unique` | Practical employee identifier; unique constraint catches duplicates. |
| `employmentType` | enum | Distinguishes full-time vs contractor — important context for any salary average. |
| `hireDate` | `date?` | Enables tenure metrics; nullable so seeding/partial records are valid. |
| `createdAt` / `updatedAt` | `timestamp` | Standard audit columns; `updatedAt` auto-bumps on write. |

### Why store `firstName` + `lastName`, not "full name"?

The assignment says full names are *generated by combining* `first_names.txt` and
`last_names.txt`. Storing the two parts (rather than a single concatenated string):

- preserves the source data faithfully (the combination is reversible),
- lets the UI sort by last name, and search either part,
- and the **full name is a trivial derivation** (`firstName + " " + lastName`) — there's no
  reason to denormalize it into its own column.

## Storing money: integer minor units

`salary` is an **`integer` of minor units** (e.g. `7500000` = 75,000.00). This is the standard
way to avoid floating-point rounding errors in compensation data.

- ❌ `float`/`real` — `0.1 + 0.2 !== 0.3`; never store money in binary floating point.
- 🟡 `numeric(12,2)` — exact, but arrives in JS as a *string* and needs care in arithmetic.
- ✅ `integer` minor units — exact, fast to `SUM`/`AVG`, trivially serializable. Formatting to
  "$75,000.00" is a presentation concern handled by `Intl.NumberFormat` in the UI.

A 32-bit `integer` maxes at ~21.4M major units — comfortably above any salary. If that ever felt
tight we'd switch to `bigint`, but `integer` keeps aggregates cheap.

`★ Insight ─────────────────────────────────────`
- **Normalize the grouping keys, not the descriptive text.** `countryCode` is constrained to
  ISO alpha-2 precisely because it's a `GROUP BY` key — inconsistent country strings would
  silently split "Germany" into multiple buckets and corrupt every per-country insight.
- **Indexes are derived from the queries, not guessed.** The composite
  `(country_code, job_title)` index exists for exactly one query — "average salary for a job
  title in a country" — letting Postgres satisfy the filter from the index. See
  [Salary Metrics](/reference/salary-metrics/) for the matching queries.
`─────────────────────────────────────────────────`

## Indexing strategy

| Index | Serves |
| --- | --- |
| `employee_country_idx (country_code)` | Per-country min/max/avg, headcount-by-country. |
| `employee_country_job_idx (country_code, job_title)` | Avg salary for a job title **in** a country (left-prefix also covers country-only). |
| `employee_job_idx (job_title)` | Cross-country "top paying roles", job-title filter without a country. |

At 10k rows Postgres could sequential-scan in milliseconds, so indexes are not strictly required
for *correctness or speed today*. They're included because (a) the persona implies real-world
growth beyond 10k, and (b) they make the intended access patterns explicit and self-documenting.
This trade-off is recorded in [Decisions](/explanation/decisions/).

## Relationship to auth

The `employee` table is **independent** of the Better-Auth `user` table. `user` represents the
*HR manager logged into the app*; `employee` represents the *people whose salaries are managed*.
They are different entities and intentionally not joined.
