The assignment is explicit: **engineers run this regularly, so performance matters.** The script
inserts 10,000 employees, with full names generated by combining `first_names.txt` and
`last_names.txt`.

## Design goals

1. **Fast** — seed 10k rows in well under a second of DB time.
2. **Deterministic** — a seeded RNG so runs are reproducible (great for tests and demos).
3. **Idempotent** — safe to re-run; clears the table first.
4. **Realistic** — varied countries, job titles, departments, and a plausible salary spread.

## Name generation

```ts
import { readFileSync } from "node:fs";

const firstNames = readFileSync("first_names.txt", "utf8").trim().split("\n");
const lastNames  = readFileSync("last_names.txt", "utf8").trim().split("\n");

// full name = `${first} ${last}` — combination of the two source lists
const fullName = (rng: () => number) =>
  `${pick(firstNames, rng)} ${pick(lastNames, rng)}`;
```

With even a few hundred entries per list, the cross-product yields far more than 10k unique
combinations, so the roster looks natural without collisions mattering.

## The performance lever: batched multi-row inserts

The naïve approach — 10,000 individual `INSERT` statements — means 10,000 network round-trips.
That dominates the runtime. The fix is **multi-row inserts inside one transaction**:

```ts
import { db } from "./index";
import { employee } from "./schema/employee";

const TOTAL = 10_000;
const CHUNK = 1_000;            // rows per INSERT statement

const rows = Array.from({ length: TOTAL }, () => makeEmployee(rng));

await db.transaction(async (tx) => {
  await tx.delete(employee);                       // idempotent reset
  for (let i = 0; i < rows.length; i += CHUNK) {
    await tx.insert(employee).values(rows.slice(i, i + CHUNK));
  }
});
```

### Why chunk at ~1,000?

PostgreSQL's wire protocol caps a statement at **65,535 bind parameters**. With ~11 columns per
row, the hard ceiling is ~5,900 rows/statement. A **1,000-row chunk (~11k params)** stays safely
under that, keeps each statement's memory modest, and captures essentially all the round-trip
savings — going bigger yields diminishing returns.

```
10,000 single inserts     → 10,000 round-trips      (slow: network-bound)
10 × 1,000-row inserts     →     10 round-trips      (fast: ~1000× fewer)
1 transaction wraps them   → one fsync at commit     (durability without per-row cost)
```

`★ Insight ─────────────────────────────────────`
- **The bottleneck is round-trips, not rows.** Inserting 10k rows is trivial work for Postgres;
  the cost is the 10k separate request/response/fsync cycles. Batching collapses those into ~10,
  which is the single biggest win and why the script feels instant.
- **One transaction = one durable commit.** Wrapping the batches in a transaction means Postgres
  flushes to disk once at `COMMIT` instead of after every statement — and it makes the seed
  atomic: a failure leaves the table in its previous state, never half-seeded.
`─────────────────────────────────────────────────`

## Determinism

A small seeded PRNG (e.g. `mulberry32`) replaces `Math.random()`. Same seed → identical roster
every run. This makes demos repeatable and lets tests assert on known generated data.

```ts
function mulberry32(seed: number) {
  return () => {
    seed |= 0; seed = (seed + 0x6d2b79f5) | 0;
    let t = Math.imul(seed ^ (seed >>> 15), 1 | seed);
    t = (t + Math.imul(t ^ (t >>> 7), 61 | t)) ^ t;
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  };
}
```

## Salary realism

Salaries are drawn so insights are interesting: a base band per job title, jittered, with a
right-skew (a few high earners) so **mean and median visibly differ** — which makes the
median metric in [Salary Metrics](/reference/salary-metrics/) demonstrably useful.

## If we needed to go faster

At 10k, batched inserts are already sub-second, so this is intentionally where we stop. For
100k–1M rows the next step is PostgreSQL **`COPY`** (binary stream), the fastest bulk-load path —
~5–10× faster than multi-row `INSERT`. It's noted as a 🔭 considered option in
[Decisions](/explanation/decisions/) but not needed for the stated scale (YAGNI).
