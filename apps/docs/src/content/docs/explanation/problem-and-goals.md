## The problem

Build a **minimal yet usable salary management tool** for an organization with
**10,000 employees**. It must be a fully functional, end-to-end product: backend, database,
and UI — not a prototype or a set of disconnected scripts.

## The user: an HR Manager

Every design decision is anchored to one persona.

> **Priya, HR Manager.** She owns the employee roster and compensation data for the whole org.
> She is comfortable with spreadsheets and admin tools but is **not** an engineer. She needs to
> add/correct employee records quickly, and she needs **answers to compensation questions** —
> "What's the average salary for a Software Engineer in Germany?" — without exporting CSVs or
> writing formulas.

What this persona implies for the build:

- **Speed at 10k rows.** Lists must paginate and filter server-side; she can't scroll 10,000 rows.
- **Trustworthy numbers.** Aggregates must be correct and computed in the database, not approximated client-side.
- **Forgiving editing.** Create/update/delete with validation and clear feedback; mistakes are recoverable.
- **No jargon.** The UI speaks in "country", "job title", "salary" — not table names.

## Functional requirements

### Employee management ✅ requirement

- Add, view, update, and delete employees through the UI.
- An employee has at minimum: **full name, job title, country, salary**.
- Capture additional meaningful fields where they help the persona (see [Data Model](/reference/data-model/)).

### Salary insights ✅ requirement

- **Min / max / average** salary of employees in a country.
- **Average** salary for a given **job title within a country**.
- Any other metrics that genuinely help an HR manager (median, headcount, distribution,
  top-paying roles — see [Salary Metrics](/reference/salary-metrics/)).

### Seeding ✅ requirement

- A seed script that inserts **10,000 employees**.
- Full names are generated by combining `first_names.txt` and `last_names.txt`.
- The script is run **regularly**, so its **performance matters** (see [Seeding](/how-to/seed/)).

## Non-functional goals

| Goal | How we meet it |
| --- | --- |
| **Correctness** | Aggregations computed in PostgreSQL; inputs validated with Zod end-to-end. |
| **Performance at 10k** | Server-side pagination + filtering, targeted indexes, batched seeding. |
| **Type safety** | oRPC gives one source of truth from DB → server → client. |
| **Maintainability** | Monorepo with clear package boundaries; Biome/Ultracite enforced. |
| **Testability** | Pure aggregation/generation logic, fast deterministic unit tests. |

## Explicit non-goals

- **Multi-tenancy / org hierarchy.** One organization, one roster.
- **Payroll execution.** We manage compensation *data*, we do not run payroll, tax, or payments.
- **Historical compensation / audit trail.** Out of scope for the minimal tool (noted as future work).
- **Role-based access control.** A single authenticated HR-manager role; no fine-grained permissions.

These are deliberate scope cuts to keep the tool *minimal yet usable*. See
[Decisions & Trade-offs](/explanation/decisions/) for the reasoning.
